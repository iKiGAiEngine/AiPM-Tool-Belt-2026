COMPLETE IMPLEMENTATION SPEC: Viewer Role Read-Only Enforcement
GOAL
Add a viewer role to the application that can log in and browse all pages/data but cannot perform any write operations. Every write action must be blocked at both the backend (HTTP 403) and the frontend (disabled buttons, guarded handler functions, visible "read-only" banner).

PART 1 — DATABASE
The users table must have a role column. Ensure this SQL is run against the production/main database:

-- Add role column if it doesn't exist
ALTER TABLE users ADD COLUMN IF NOT EXISTS role text NOT NULL DEFAULT 'user';
-- Create the viewer account (adjust email/password hash as needed)
-- Use your existing OTP or password system to create this user,
-- then set its role:
UPDATE users SET role = 'viewer' WHERE email = 'viewer@yourdomain.local';
If you use a role enum instead of free-text, add 'viewer' to the enum first.

PART 2 — BACKEND MIDDLEWARE
2A. Create requireWriteAccess middleware
In your server middleware file (wherever requireAuth or session checks live), add:

export function requireWriteAccess(req: Request, res: Response, next: NextFunction) {
  const user = (req.session as any)?.user;
  if (!user) return res.status(401).json({ error: "Unauthorized" });
  if (user.role === "viewer") {
    return res.status(403).json({ error: "READ_ONLY", message: "Viewer accounts cannot make changes." });
  }
  next();
}
2B. Apply middleware to ALL write routes
In your server/routes.ts (or equivalent), add requireWriteAccess as middleware to every POST, PUT, PATCH, and DELETE route. Examples:

// Before:
app.post("/api/proposal-log", requireAuth, async (req, res) => { ... });
// After:
app.post("/api/proposal-log", requireAuth, requireWriteAccess, async (req, res) => { ... });
Apply this to every single write route in the application — proposal log, spec extractor, plan parser, quote parser, project start, settings, templates, admin endpoints, notifications, backups, everything.

PART 3 — AUTH CONTEXT (FRONTEND)
3A. Locate your useAuth hook
Find the file that exports useAuth (typically client/src/lib/auth.ts or client/src/hooks/useAuth.ts).

3B. Add isViewer to the returned object
The hook currently returns something like { user, isAdmin, ... }. Add:

const isViewer = user?.role === "viewer";
// Return it alongside the existing fields:
return { user, isAdmin, isViewer, /* ...other existing fields */ };
This single change propagates everywhere useAuth() is called.

PART 7 — KEY TECHNICAL NOTES
The isViewer flag must come from the same useAuth() hook used everywhere else. Do not create a separate context or prop-drill it.
Sub-components: If a page has inner sub-components (functions defined inside the same file that are NOT React hooks), each one that needs isViewer must call useAuth() independently — you cannot pass it down from an outer function that is not a React component.
TypeScript: After each file change, run npx tsc --noEmit and verify the error count has not increased. All edits are additive and should not introduce new type errors.
Do not touch: drizzle.config.ts, vite.config.ts, server/vite.ts, package.json. Do not install new packages — this implementation requires zero new dependencies.
Toast calls: Use the existing useToast() hook already present in each file. Do not import a different toast system.
Handler deps arrays: When adding isViewer to a useCallback, add it to the dependency array if the linter requires it.
